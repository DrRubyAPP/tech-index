Research Participation — N=1 Longitudinal Study
Be part of the first N=1 women's
healthspan study. Led by Dr. Brenner.
We are recruiting women to participate in a 6-month double-blind N=1 supplement and skin tracking study. Your longitudinal data will help establish scientifically defensible, individualized healthspan baselines — and improve recommendations for every DrRuby user.
Study Protocol · 6-Phase Double-Blind Design
PHASE 1 · Week 0–2
Baseline Establishment
Skin scan baseline · Lab panel · Supplement audit · Lifestyle questionnaire. No intervention.
PHASE 2 · Week 2–6
Washout Period
Participant stops current supplements. Weekly skin scans continue. Stabilize baseline signal.
PHASE 3 · Week 6–14
Blinded Intervention A
Participant receives either active supplement stack or placebo (double-blind). Daily scan + weekly self-report.
PHASE 4 · Week 14–16
Second Washout
Crossover preparation. Continuation of scans to capture return to baseline.
PHASE 5 · Week 16–24
Blinded Intervention B
Crossover: participants who had active now receive placebo, and vice versa. Same scan + report cadence.
PHASE 6 · Week 24–26
Unblinding + Debrief
Results revealed to participant. Personalized N=1 report delivered. Dr. Brenner debrief session available.

⚙ Engineering Notes — Skin Scan UX Rules
Frequency Gate (hard rules)
· Hard cooldown: 72h minimum between any scans (cannot be overridden)
· Default mode: monthly cadence (30-day soft prompt)
· Tracking Mode: user-initiated, enables weekly scans, max 12 weeks, auto-expires
· If user attempts 3+ scans in 7 days → show gentle prompt directing to dermatologist
· Rationale: prevent health anxiety loops; protect Insight Engine data quality
Image Quality Gate (technical)
· Run on-device, real-time (target <200ms per frame)
· V1 (MVP): Laplacian variance (blur), histogram (exposure), bounding-box ratio (coverage), inter-frame delta (motion) — all rule-based, Core Image / CameraX
· V2: lightweight MobileNet on-device classifier — after V1 baseline
· Capture button DISABLED until all 4 checks pass
· Error messages must be specific: not "bad photo" — "Too dark, move to better light" / "Too blurry, hold still" / "Frame the skin closer"
· Only high-quality images enter Insight Engine; bad images rejected, not stored

⚡ DrRuby AI — 4 Design Principles (Chatbot as UI, not product)
1 · Proactive push
Insight Engine detects a change → chatbot sends a message. User doesn't need to ask first. Data drives the conversation.
2 · LLM = translator only
LLM receives structured output from Insight Engine (Layer 3) and converts it to natural language + citation. It does not generate insights itself.
3 · Chat = data logging
"I started magnesium today" → auto-logged to Intervention Graph, tracking begins. No forms. Conversation is the input interface.
4 · Hybrid UI
Score cards at top are clickable → chatbot explains that specific score using the user's own longitudinal data. Chat is the explanation layer, not the entry point.

AI Technical Architecture — 五层系统架构
The product is the Insight Engine (Layer 3). The chatbot is the UI. RAG knowledge base is table stakes — not the moat.
Layer 1
Evidence Graph · RAG Knowledge Base
Structured scientific claims from Brenner (NAD+, metabolism), Gunter (women's health, pending), dermatology literature · evidence_level tiers: rct / clinical_guideline / expert_opinion / emerging · PostgreSQL + pgvector · Not the moat — table stakes
Table stakes
↓
Layer 2
User Longitudinal Data · On-Device
Labs, wearables, skin photos, symptoms, interventions, outcomes — stored locally encrypted (Core Data / Room + SQLCipher) · Tiered consent: Tier 0 default on-device, Tier 1 opt-in encrypted upload, Tier 2 opt-in research contribution
Input to moat
↓
Layer 3
DrRuby Insight Engine · N-of-1 Analytics ← THE MOAT
Change detection (moving average, z-score) · Intervention Graph (what worked for this user, with lag and confidence) · Proprietary scores: Skin Age Score, Metabolic Score, Recovery Score, Inflammation Score, Hormonal Stability Index · V1: rule-based + statistical · V2: Bayesian attribution · Patent candidates: scoring methodology + Intervention Graph structure
IP · File patent
↓
Layer 4
Recommendation Engine
Combines Layer 3 insights + Layer 1 evidence → structured recommendation object · This runs before the LLM, not inside it · Output is structured data, not prose
Differentiator
↓
Layer 5
Conversation Interface · UI Only
LLM API (Claude / GPT-4o) translates Layer 3+4 structured output into natural language + citations · Proactive push when Insight Engine detects change · Conversational data logging ("I started NR today" → writes to Intervention Graph) · Hybrid UI: score cards + chat · FDA guardrails embedded in system prompt
UI, not product
Why this can't be replicated in 3 weeks: Layer 1 (RAG on PubMed) can. Layers 3–4 cannot — because they require months of a user's longitudinal data and the Intervention Graph that grows with it. The data is the moat. Brenner's role is Scientific Advisory Board — he defines scoring methodology, not claim validation.
